1. Workout Sessions Table Structure
Every time a user performs a workout, log a session with:

Field	Description
PK	USER#<sub>
SK	WORKOUT_SESSION#<date>#<uuid>
WorkoutId	ID of the workout performed
WorkoutName	Name of the workout
Date	ISO date from user's device
Notes	Optional notes (e.g., "felt strong")
GSI1PK	USER#<sub>
GSI1SK	DATE#<date>#<timestamp>
2. Exercise Performance Logs
Each session has multiple performed exercises, like:

Field	Description
PK	USER#<sub>
SK	WORKOUT_SESSION#<date>#<uuid>#EXERCISE#<uuid>
ExerciseName	e.g., "Bench Press"
SetsPerformed	[{ set: 1, reps: 10, weight: 135 }, ...]
WorkoutId	Links to definition
WorkoutSessionId	Link to parent session

3. Session Ingestion
Your frontend would:

Choose a workout from the definitions

Render each exercise and allow the user to log sets, reps, weight

Submit the full session to a POST /workout-tracking endpoint

Storage Plan:
DynamoDB table WorkoutSessions:

PK: USER#<sub>

SK: WORKOUT_SESSION#<date>#<uuid> (parent)

SK: WORKOUT_SESSION#<date>#<uuid>#EXERCISE#<uuid> (children)

Use GSI1 for querying by date across sessions.
